## 2. アナリスト・ナレッジベースの構造

### 2.1 4層アーキテクチャ

```
                    抽象度 高
                        ▲
                        │
Layer 1: dogma.md ──────┤ 投資哲学（5原則 + 確信度スケール + 却下基準）
                        │ → 「何が競争優位性で、何がそうでないか」の根本思想
                        │
Layer 2: KB1 (8ルール) ─┤ 演繹的チェックリスト
                        │ → 「全主張に対してこの8つを機械的に確認せよ」
                        │
Layer 3: KB2 (12パターン)┤ 帰納的パターンカタログ
                        │ → 「よくある誤り(A-G)とよくある高評価(I-V)のパターン」
                        │
Layer 4: KB3 (5社34件) ─┤ 具体例（暗黙知のアンカー）
                        │ → 「Y が実際にどう判断したか」の一次データ
                        ▼
                    抽象度 低
```

### 2.2 KB のバリアント

|-----------|------|------|------|

| **トランスクリプト版** (KB1-T/KB2-T/KB3-T) | `analyst/transcript_eval/` | CA Strategy（決算説明会評価） | +rule12（一次情報判定）、Seven Powers 参照 |
| **MAS版** (未定義) | — | MAS Investment Team | **要設計** |

### 2.3 確信度スコアの分布（Y の判断パターン）

```
  確信度   割合     典型的な状況
  ───────  ──────   ─────────────────────────────────
  90%      6%  ██   構造的優位性 + 明確なCAGR接続 + 定量的裏付け
  70%     26%  ████████████  妥当な仮説 + 一定の裏付け
  50%     35%  █████████████████  方向性OK、裏付け不十分（最頻値）
  30%     26%  ████████████  飛躍的解釈・因果関係の逆転

```

**MAS への示唆**: AI は過信しやすい。50% が最頻値であり、90% は極めてまれ。

## 3. MAS アーキテクチャにおける KB の位置づけ

### 3.1 MAS の 5 フェーズと KB の接点

```
Phase 0: POSTMORTEM（失敗パターン分析）
  └── KB接点: なし（定量ベース）

Phase 1: SCREENING（500→50-80銘柄）
  └── KB接点: なし（Pythonファクタースコア）

Phase 2: ANALYSIS（4並列エージェント）
  ├── [T2] Fundamental Analyst ←── 🔴 KB統合ポイント①
  │     競争優位性の存在・質を評価する際に dogma + KB1 を参照
  ├── [T3] Valuation Analyst
  │     KB接点: 限定的（CAGR接続の評価時に dogma 参照の可能性）
  ├── [T4] Sentiment Analyst ←── 🔴 KB統合ポイント②
  │     トランスクリプト分析時に KB1-T/KB3-T を参照
  └── [T5] Macro/Regime Analyst
        KB接点: なし

Phase 3: DEBATE（Bull vs Bear）
  ├── [T6] Bull Advocate ←── 🟡 KB統合ポイント③
  │     投資論拠の構築時に KB2（高評価パターン I-V）を参照
  └── [T7] Bear Advocate ←── 🟡 KB統合ポイント④
        反論構築時に KB2（却下パターン A-G）を参照

Phase 4: DECISION
  └── [T8] Fund Manager ←── 🔴 KB統合ポイント⑤
        最終判断時に dogma（確信度スケール）を参照
        既存判断への警鐘機能に KB2 を活用

Phase 5: PORTFOLIO CONSTRUCTION
  └── KB接点: なし（Pythonベース最適化）
```

### 3.2 KB 統合ポイントの詳細

#### 🔴 統合ポイント①: Fundamental Analyst

| 項目 | 内容 |
|------|------|
| **参照する KB** | dogma.md + KB1（8ルール） |
| **用途** | SEC Filings（10-K/10-Q）から抽出した事実を「競争優位性」として評価する際の判断基準 |
| **期待される効果** | 結果と原因の取り違え防止、戦略と優位性の混同防止 |
| **統合方法** | システムプロンプトに dogma の要約 + KB1 ルールリストを注入 |
| **リスク** | コンテキスト増大（dogma ~12KB + KB1 ~15KB = ~27KB） |

#### 🔴 統合ポイント②: Sentiment Analyst

| 項目 | 内容 |
|------|------|
| **参照する KB** | KB1-T（9ルール）+ KB3-T（5社 few-shot） |
| **用途** | 決算トランスクリプトから競争優位性に関連するシグナルを評価 |
| **期待される効果** | 経営陣の自画自賛と構造的優位性の区別 |
| **統合方法** | CA Strategy の transcript-claim-extractor パターンを流用 |
| **リスク** | CA Strategy で Phase 1 が 0 件抽出のブロッカーが未解決 |

#### 🟡 統合ポイント③④: Bull/Bear Advocate

| 項目 | 内容 |
|------|------|
| **参照する KB** | KB2 パターン集（Bull→高評価 I-V、Bear→却下 A-G） |
| **用途** | ディベートの論拠構築を Y の判断パターンに沿って行う |
| **期待される効果** | 表面的な議論ではなく、構造的な優位性の有無に焦点を当てたディベート |
| **統合方法** | プロンプトに KB2 のパターン名とトリガー条件を注入 |
| **リスク** | Bull が KB2(I-V) を好意的に使い、Bear が KB2(A-G) を批判的に使うと偏向する可能性 |

#### 🔴 統合ポイント⑤: Fund Manager

| 項目 | 内容 |
|------|------|
| **参照する KB** | dogma（確信度スケール） + KB2（パターン全体） |
| **用途** | 確信度スコア（0-100）の基準を Y の判断軸に整合させる |
| **期待される効果** | (a) 過信の抑制（50% が最頻値）、(b) 既存判断への警鐘 |
| **統合方法** | dogma の確信度スケールを FM のスコアリング基準に組み込み |
| **リスク** | Y の 5 段階（10/30/50/70/90%）と FM の 100 段階（0-100）のマッピング |

## 4. 統合の設計方針

### 4.1 3つのアプローチ比較

| アプローチ | 説明 | メリット | デメリット |
|-----------|------|---------|-----------|
| **A. 全注入** | 全エージェントに dogma + KB 全体を注入 | 判断の一貫性が最大 | コンテキスト肥大（~62KB×10エージェント）、コスト増 |
| **B. 層別注入** | エージェントの役割に応じて必要な KB 層のみ注入 | コスト効率的、役割明確 | エージェント間の判断基準にズレが生じうる |
| **C. KB エージェント分離** | 専用の「KB Evaluator」エージェントを新設し、他エージェントの出力を KB で再評価 | 関心の分離、KB更新が容易 | パイプラインが長くなる、レイテンシ増 |

### 4.2 推奨: B+C のハイブリッド

## 6. 設計上の重要な判断

### 6.1 Y の確信度（5段階）と FM の確信度（100段階）のマッピング

**問題**: Y は 10/30/50/70/90% の 5 段階。MAS の FM は 0-100 の連続値。

**提案**: FM のスコアリングに 2 つの軸を導入

```
┌──────────────────────────────────────────────────────┐
│ Fund Manager のスコアリング構造                        │
│                                                      │
│  軸1: 競争優位性の確信度（Y スケール準拠）             │
│       dogma の 5 段階を直接参照                        │
│       → CA Evaluator の出力を使用                     │
│                                                      │
│  軸2: 投資確信度（FM 独自判断）                        │
│       バリュエーション + マクロ + センチメント          │
│       → 0-100 の連続スコア                            │
│                                                      │
│  最終判断 = f(軸1, 軸2)                               │
│  ※ 軸1 が 30% 以下なら、軸2 が高くても PASS を推奨    │
└──────────────────────────────────────────────────────┘
```

### 6.2 KB の更新とバージョン管理

**問題**: バックテスト期間中に KB が更新されると、先読みバイアスに類似する問題が発生する。

**方針**:
- バックテスト全期間で**固定バージョン**の KB を使用
- KB のバージョンは `config.json` に記録
- Y のフィードバックによる KB 更新は、次回のバックテスト実行から適用

### 6.3 CA-Eval 実績の活用

CA-Eval で 12 社の評価実績がある。この知見を MAS に活用する方法:

| CA-Eval の知見 | MAS への適用 |
|---------------|-------------|
| 12 ルールの適用率分布 | CA Evaluator のプロンプト設計に反映 |
| Y のスコアとの乖離パターン | KB 要約の重点項目を決定 |
| AI 批判プロセス（ドラフト→批判→反映） | CA Evaluator に 2 段階評価を導入 |
| 全ルール明示の透明化 | `decision_log.json` に KB ルール適用結果を含める |
