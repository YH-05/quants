# Numerai -- Skills, MCP, 自律型 AI リサーチエージェントの分析レポート

**作成日**: 2026-02-25
**対象**: Numerai の Skills/MCP アプローチと MAS4InvestmentTeam への適用可能性

---

## 1. Numerai の概要とビジョン

### 1.1 「世界最後のヘッジファンド」

Numerai は Richard Craib が創設した、従来のヘッジファンドとは根本的に異なるアーキテクチャを持つ投資運用組織である。その核心は「世界中のデータサイエンティストの集合知を、一つの Meta Model に統合する」という構造にある。従来のヘッジファンドが社内に少数の優秀なポートフォリオマネージャーやクオンツを雇用し、彼らの知的リソースに依存する形で運用するのに対し、Numerai は外部のコントリビューター（参加者）が提出する予測モデルを統合することで、単一組織の知的限界を超えるアプローチをとっている。

2024年の実績として 25.45% のリターンを達成し、Sharpe Ratio は 2.75 を記録した。この成果は、JPMorgan Asset Management から最大 $500M（約750億円）の投資を引き出すに至っている。従来型のクオンツファンドやファンダメンタルファンドと比較しても、このパフォーマンスは業界上位に位置する。

### 1.2 Meta Model とステーキングメカニズム

Numerai のプラットフォームは、参加者が予測モデルを提出し、その予測の品質に応じて報酬を得る（あるいはステークを失う）仕組みで成り立つ。具体的には、参加者は暗号通貨 NMR（Numeraire）をステーク（担保として供託）し、自らの予測に「スキン・イン・ザ・ゲーム」を持つ。予測が正確であればステークに応じた報酬を得、不正確であればステークの一部を失う。この経済的インセンティブ設計により、参加者は無責任な予測を提出する動機を失い、真に自信のあるモデルのみが提出される構造になっている。

提出された多数の予測モデルは、Numerai 側で Meta Model に統合される。個々のモデルの弱点を相互に補完し合うことで、単一のモデルでは到達し得ない予測精度を実現する。これはアンサンブル学習の大規模版とも言えるが、モデルの多様性が自然に担保される点で、社内チームが同じデータ・同じ文化の中で作るアンサンブルとは質的に異なる。

### 1.3 ヘッジファンド産業の構造的問題への挑戦

Craib は Numerai の最終的なビジョンを以下のように述べている。

> "I really think that there are pods at Millennium that are paid $100 million a year with code that could be replicated in 40 hours by Claude."

> "We want fewer human beings in the hedge fund management industry."

これは単なる挑発的発言ではなく、ヘッジファンド産業が抱える構造的問題への直接的な指摘である。マルチストラテジーファンド（Millennium, Citadel, Point72 等）は、各ポッド（運用チーム）に年間数千万ドルから1億ドル規模の報酬を支払っている。しかし、そのコードの多くは比較的標準的なファクターモデルやシグナル処理であり、最先端の AI エージェントであれば数十時間で再現可能だという主張である。

この主張の当否はともかく、人間のアナリスト/PM に依存する運用モデルのコスト構造が、AI エージェントの急速な進化によって持続不可能になりつつあるという認識は、業界内でも広がっている。Numerai はこの構造変化の最前線に立ち、「人間のデータサイエンティストが参加するプラットフォーム」から「AI エージェントが自律的に参加するプラットフォーム」への移行を明確に宣言している。

## 2. SKILLS.md の設計哲学と技術的詳細

### 2.1 問題の定義: フロンティア AI の「最初の1時間」問題

Numerai の SKILLS.md を理解するためには、まず「フロンティア AI が直面する問題」を正確に把握する必要がある。Claude、GPT、Gemini といった最先端の大規模言語モデルは、金融に関する一般的な知識を膨大に持っている。統計学、機械学習、ファイナンス理論について極めて詳しい。しかし、Numerai 固有の研究プロセス、つまり「Numerai ではどのようにクロスバリデーションをすべきか」「金融時系列の特性をどう扱うべきか」「どのような特徴量が Numerai のデータセットで有効か」という暗黙知を持っていない。

Richard Craib は AI Street のインタビューでこの問題を明確に言語化している。

> "No matter how smart it gets, it doesn't have Numerai Skills, which is the skills.md file we made. So, when you ask it, do a whole bunch of research on Numerai, sort of within the first hour, it'll make three bad mistakes. It's not really its fault, it's not because it's dumb, it just doesn't know quite how we do things. And so, once it's got access to the skills.md, it'll be like, oh, well, if I need to do that, I'm just -- I have to use the skills way of doing it."

この発言は3つの重要な洞察を含んでいる。

**第一に、知能と専門知識は別物である。** AI がどれだけ賢くても、ドメイン固有の「やり方」を知らなければ、基本的な誤りを犯す。これは人間のアナリストが新しいファンドに入社した際に、そのファンドの投資哲学や分析プロセスを学ぶ必要があるのと構造的に同じである。

**第二に、誤りは「最初の1時間」で発生する。** つまり、AI は作業を開始した直後に、見当違いの方向に進む。この初期の誤りは、その後の全ての分析結果を汚染する。金融の研究では、最初のデータ処理や特徴量設計の段階での誤りが、最終的なモデルの性能に決定的な影響を与える。

**第三に、SKILLS.md を与えれば問題は解決する。** AI は「Numerai 流のやり方」を記述した構造化ドキュメントを読めば、即座にそのプロトコルに従って作業を実行できる。人間のアナリストが数ヶ月かけて体得する暗黙知を、AI は SKILLS.md を読むだけで即座に獲得できる。

### 2.2 SKILLS.md の構造と設計原則

Numerai の SKILLS.md は、Claude Code の SKILL.md 形式を採用している。これは偶然ではなく、意図的な選択である。Claude Code のスキルシステムは、AI エージェントに対して「何を」「どのように」「なぜ」を構造的に伝達するためのフレームワークであり、Numerai の研究プロセスの暗黙知を構造化するのに適している。

SKILLS.md がカバーする内容は以下の5領域に分類される。

**実験設計の方法論。** 金融時系列データの特性を考慮した実験の設計方法。時系列クロスバリデーション（Purged K-Fold CV, Embargo 付き CV）の適用方法、過学習を防ぐためのサンプル分割戦略、バックテスト期間の設定基準などが含まれる。一般的な機械学習の教科書に書かれているランダム分割の CV を金融データに適用すると、未来のデータが学習データに混入し（ルックアヘッドバイアス）、誤って高い性能を示すモデルが生成される。これが Craib の言う「最初の1時間での3つの誤り」の典型例である。

**特徴量のテスト手法。** 新しい特徴量（フィーチャー）を Numerai のデータセットで検証する際の方法論。特徴量の中立化（feature neutralization）、ターゲットとの相関分析（IC: Information Coefficient）、特徴量の安定性（フィーチャーの時系列的な予測力の安定度）の評価方法などが規定されている。

**結果の評価方法。** モデルの予測結果を評価する際の基準。Numerai 固有の評価指標（Correlation, TC: True Contribution, FNC: Feature Neutral Correlation）の解釈方法、ステーキング戦略の評価、Meta Model への貢献度の計測方法が含まれる。

**失敗の診断パターン。** モデルが期待通りに機能しない場合の診断フレームワーク。過学習のサイン、特徴量の陳腐化（feature decay）、レジーム変化への対応不足、データリーケージの検出方法など、経験豊富なコントリビューターが長年の参加を通じて蓄積した「失敗パターンのカタログ」が構造化されている。

**反復プロセス。** 仮説生成→実験→評価→改善のサイクルをどのように回すべきかのガイドライン。一度に大きな変更を加えるのではなく、小さな仮説を一つずつ検証する科学的アプローチが推奨されている。

### 2.3 「暗黙知の構造化」というパラダイム

SKILLS.md の設計哲学の核心は、「経験豊富な研究者の暗黙知を、AI が消費可能な形式で構造化する」というパラダイムにある。これは Michael Polanyi の暗黙知（tacit knowledge）の概念を、AI エージェント時代に再解釈したものと言える。

Polanyi は「We can know more than we can tell（我々は語り得る以上のことを知っている）」と述べたが、SKILLS.md はまさにこの「語り得ない知識」を語り得る形式に変換する試みである。経験豊富な Numerai コントリビューターは、「このデータセットではこういうアプローチがうまくいく」「この特徴量は一見良さそうに見えるが、実際にはこういう罠がある」といった暗黙知を持っている。SKILLS.md はこれを、AI エージェントのコンテキストウィンドウに注入可能な構造化テキストに変換する。

重要なのは、SKILLS.md は単なるドキュメントではなく、**AI エージェントの行動を制約し、方向づけるための実行可能な指示書**であるという点だ。人間向けのドキュメントとの違いは、曖昧さの排除と、具体的な手順の明示にある。「適切なクロスバリデーションを使用してください」ではなく、「Purged K-Fold CV を使用し、Embargo 期間は 12 営業日に設定してください。ランダム分割の CV は絶対に使用しないでください」のように、AI が解釈の余地なく従える指示が記述される。

### 2.4 SKILLS.md と AI エージェントの関係

NumerCon 2026 で Numerai は「The Era of Agents」を宣言した。これは SKILLS.md の役割が、「人間のデータサイエンティストを支援するツール」から「AI エージェントが自律的にリサーチを実行するためのフレームワーク」へと進化したことを意味する。

具体的には、以下の変化が起きている。

**従来（人間向け）**: 人間のデータサイエンティストが SKILLS.md を参考にしながら、自分でコードを書き、実験を設計し、モデルを提出する。SKILLS.md は「ガイドライン」としての役割。

**現在（エージェント向け）**: AI エージェントが SKILLS.md をシステムプロンプトとして読み込み、自律的に実験を設計し、特徴量をテストし、モデルを構築・提出する。SKILLS.md は「行動規範」としての役割。人間は AI エージェントの設計・監督に移行する。

この進化により、人間のデータサイエンティストの役割は「自分で分析する人」から「AI リサーチアシスタントを設計・監督する人」に変わりつつある。人間は投資仮説の生成、SKILLS.md の改善、エージェントの評価・監督に注力し、実際のコーディング・実験・モデル構築は AI エージェントが担う。

## 3. MCP の技術的分析とエージェント自律化

### 3.1 Model Context Protocol（MCP）の概要

Numerai MCP は、AI エージェントが Numerai プラットフォームにプログラム的にアクセスするための標準化されたプロトコルである。MCP（Model Context Protocol）は Anthropic が提唱した、AI エージェントが外部ツールやデータソースと対話するためのオープン規格であり、Numerai はこれを自社プラットフォームへのエージェント接続のインターフェースとして採用した。

MCP の導入により、AI エージェントは以下の操作をプログラム的に実行できるようになった。

**モデル作成。** エージェントが自律的に新しいモデルをプラットフォーム上に登録できる。モデルの命名、設定、パラメータの指定を API 経由で行う。

**予測のアップロード。** 学習済みモデルの予測結果を、Numerai のサブミッションシステムに直接アップロードできる。従来は人間が手動でCSVファイルをアップロードしていた作業が、エージェントの自律的なワークフローに組み込まれる。

**診断テストの実行。** アップロードされた予測の品質を評価する診断テスト（相関分析、特徴量中立化後の相関、ターゲットコンシステンシー等）をプログラム的に実行し、結果を取得できる。

**パフォーマンスデータの取得。** モデルの過去のパフォーマンス（ライブリターン、ステーキング結果、他モデルとの相関等）をリアルタイムで取得し、分析に利用できる。

### 3.2 MCP がもたらす自律リサーチループ

MCP の技術的な意義は、個々の API 操作の自動化ではなく、**フルリサーチサイクルの自律化**を可能にする点にある。SKILLS.md と MCP を組み合わせることで、以下のような完全な自律リサーチループが実現する。

```
[SKILLS.md による行動規範の注入]
    |
    v
1. 仮説生成（SKILLS.md の実験設計方法論に基づく）
    |
    v
2. 特徴量設計・テスト（SKILLS.md の特徴量テスト手法に基づく）
    |
    v
3. モデル構築・学習
    |
    v
4. 予測生成
    |
    v
5. MCP: 予測のアップロード
    |
    v
6. MCP: 診断テスト実行
    |
    v
7. 結果評価（SKILLS.md の評価方法に基づく）
    |
    v
8. 失敗診断（SKILLS.md の診断パターンに基づく）
    |
    v
9. 改善仮説生成 → 1. に戻る
```

このループにおいて、人間の介入は不要である。エージェントは SKILLS.md に基づいて「正しいやり方」を知っており、MCP を通じてプラットフォームとの全てのインタラクションを自律的に実行できる。

### 3.3 Atomic Blockchain Staking（2026年7月4日ローンチ予定）

Numerai は更に、エージェントの自律性を経済的な意思決定の領域にまで拡張しようとしている。2026年7月4日にローンチ予定の Atomic Blockchain Staking では、以下の機能が提供される。

**ウォレット選択の自由。** エージェントは特定のウォレットプロバイダーに拘束されず、任意のウォレットを使用できる。

**柔軟なステーク管理。** ステークの追加・引き出し・リアロケーションをプログラム的に実行できる。

**NMR または USDC でのステーキング。** 従来の NMR に加え、USDC でもステーキングが可能になり、暗号通貨のボラティリティリスクを回避できる。

**エージェントによるプログラム的ステーク管理。** AI エージェントが自律的に、モデルの確信度に応じてステーク額を調整できる。確信度が高いモデルにはより多くステークし、パフォーマンスが低下しているモデルのステークを減らすといった動的な資金管理が自動化される。

これにより、AI エージェントは「リサーチ→予測→ステーク管理」の全プロセスを自律的に実行できるようになる。人間の役割は、エージェントの設計とリスク管理のパラメータ設定（最大ステーク額、ドローダウン制限等）に限定される。

### 3.4 SEC EDGAR MCP との対比

当プロジェクト（MAS4InvestmentTeam）が使用している SEC EDGAR MCP と Numerai MCP は、設計思想において多くの共通点を持つ。

| 項目 | Numerai MCP | SEC EDGAR MCP（当プロジェクト） |
|------|-------------|-------------------------------|
| **データソース** | Numerai プラットフォーム（匿名化金融データ） | SEC EDGAR（公開企業の Filings） |
| **主要操作** | モデル作成、予測アップロード、診断、パフォーマンス取得 | 企業情報取得、Filings 取得、財務データ抽出、インサイダー取引 |
| **エージェント連携** | SKILLS.md + MCP = 自律リサーチ | KB + MCP = 自律ファンダメンタル分析 |
| **先読みバイアス対策** | データセット側で管理（Numerai がデータを制御） | PointInTimeDataManager + 時間的制約プロンプト |

両者の根本的な類似性は、「AI エージェントが外部データソースとプログラム的にインタラクションし、Skills/KB に基づいて正しい分析プロセスを実行する」というアーキテクチャにある。

## 4. Faith データセットと LLM 特徴量の意味

### 4.1 Faith データセット: LLM 駆動の特徴量の初導入

NumerCon 2026 で発表された Faith データセットは、Numerai の歴史において画期的な転換点である。従来の Numerai データセットは、数値化された金融指標（価格ベースのファクター、財務比率、テクニカル指標等）で構成されていた。Faith データセットは、これに加えて LLM（大規模言語モデル）が生成した特徴量を初めて導入した。

LLM 特徴量とは、企業の非構造化テキストデータ（決算説明会トランスクリプト、10-K/10-Q テキスト、ニュース記事等）を LLM で処理し、数値化した特徴量である。具体的には以下のような情報が特徴量化されていると考えられる。

**経営者のセンチメント。** 決算説明会での経営者の発言トーン、使用する言葉の変化、将来見通しに対する自信度合い。

**事業戦略の構造的分析。** 10-K のビジネスセクション（Item 1）や MD&A（Item 7）のテキストから抽出された、事業の構造的特性、競争環境、リスク要因の変化。

**テキストの複雑性・可読性。** Filings のテキストの複雑性（Fog Index, Flesch Reading Ease 等）。学術研究では、Filings のテキストが複雑になるほど、企業が悪いニュースを隠蔽している傾向があることが示されている。

**クロスリファレンス・シグナル。** 経営者の発言と財務データの一致/不一致、前四半期の発言と実績の整合性。

### 4.2 LLM 特徴量の投資的意味

Faith データセットの導入は、金融における AI の役割を根本的に変える可能性を持つ。

**第一に、情報処理の次元が拡大する。** 従来のクオンツモデルは数値データのみを処理していた。LLM 特徴量の導入により、テキストデータという巨大な情報源がクオンツモデルの入力に加わる。年間数万件の決算説明会トランスクリプト、数十万ページの SEC Filings を人間が全て読むことは不可能だが、LLM はこれを網羅的に処理できる。

**第二に、非効率性の発見領域が広がる。** 市場の効率性仮説（EMH）は数値情報については概ね成立するが、非構造化テキスト情報の処理においてはまだ非効率性が残っているとする研究が増えている。LLM 特徴量は、この非効率性を体系的に捕捉するツールとなる。

**第三に、Numerai のエコシステムの参入障壁が変化する。** 従来は統計学やプログラミングのスキルが Numerai への参入障壁だった。LLM 特徴量の導入により、テキスト分析の専門知識や、AI エージェントを設計・監督する能力が新たな差別化要因になる。

### 4.3 2セット目・3セット目の LLM 特徴量

Numerai は既に2セット目の LLM 特徴量をリリースし、3セット目を開発中である。これは LLM 特徴量の有用性が初期のテストで確認されたことを示している。

複数セットの LLM 特徴量が存在する理由は、以下のように推測される。

- **異なる LLM を使用した多様性の確保**: GPT, Claude, Gemini 等、異なる LLM が異なるバイアスを持つため、複数の LLM から生成された特徴量はアンサンブル効果を生む
- **異なるテキストソースの活用**: 決算トランスクリプト、SEC Filings、ニュース記事など、ソースごとに異なるセットを生成
- **時系列的な更新**: LLM 自体の進化に伴い、新しいモデルで再生成された特徴量が追加される

### 4.4 当プロジェクトとの接点

Faith データセットの概念は、当プロジェクトの CA Strategy（競争優位性ベース投資戦略）と直接的に関連する。CA Strategy の Phase 1（transcript-claim-extractor）は、まさに LLM を使って決算トランスクリプトから「競争優位性に関する主張」を抽出するプロセスであり、これは Faith データセットの特徴量生成プロセスと本質的に同じである。

違いは、Numerai が数値特徴量に変換するのに対し、当プロジェクトは構造化されたテキスト（Claim + 確信度スコア）として保持する点にある。この違いは、Numerai が「統計的アンサンブル」を志向し、当プロジェクトが「説明可能な投資判断」を志向しているという、設計思想の差に起因する。

## 5. MAS4InvestmentTeam への適用

### 5.1 SKILLS.md パターンの mas-invest Skills への応用

Numerai の SKILLS.md が解決する問題と、当プロジェクトの `.claude/skills/` 体系が解決する問題は構造的に同一である。両者とも「フロンティア AI が持たないドメイン固有の暗黙知を、構造化テキストとして注入する」ことで、AI エージェントの初期行動品質を劇的に向上させるアプローチをとっている。

当プロジェクトの現在の Skills 体系と Numerai SKILLS.md の対応関係を整理する。

| Numerai SKILLS.md の要素 | 当プロジェクト対応物 | 現状 | 改善機会 |
|-------------------------|---------------------|------|---------|
| 実験設計の方法論 | `backtesting-guide-lookahead-bias-elimination.md` | 先読みバイアスに焦点 | 実験設計全般への拡張 |
| 特徴量テスト手法 | CA Strategy KB1-T（9ルール） | 競争優位性の主張抽出に特化 | 汎用的な特徴量評価基準への発展 |
| 結果の評価方法 | `strategy.risk.calculator` + `ca_strategy.evaluator` | Python コードとして存在 | Skills として AI エージェントが参照可能な形式に |
| 失敗の診断パターン | KB2（12パターン: 却下A-G + 高評価I-V） | 競争優位性の評価パターンのみ | 投資判断全般の失敗パターンへ拡張 |
| 反復プロセス | CA-Eval のドラフト→批判→反映サイクル | 実装済み | MAS 全体の反復プロセスとして汎用化 |

Numerai から学ぶべき最大の教訓は、**SKILLS.md を「ドキュメント」ではなく「実行可能な行動規範」として設計する**ことの重要性である。当プロジェクトの dogma.md や KB1/KB2/KB3 は内容的には優れているが、現状では「アナリストの判断基準の記録」という位置づけに留まっている。これを「AI エージェントが読み込んだ瞬間に、その基準に従って行動を開始できる形式」に昇格させることが、Numerai パターンの最重要の適用ポイントである。

### 5.2 KB（dogma + KB1/KB2/KB3）の SKILLS.md 化

当プロジェクトのナレッジベースは、既に Numerai SKILLS.md と同等の構造を持っている。以下に、KB の各層を SKILLS.md 形式に変換する具体的な方針を提示する。

#### dogma.md → Investment Philosophy Skill

dogma.md は投資哲学（5原則 + 確信度スケール + 却下基準）を定義しており、これは Numerai SKILLS.md における「実験設計の方法論」に相当する。SKILLS.md 化のポイントは以下の通り。

- **判断の前提条件を明示する**: 「まず dogma の5原則を確認し、対象企業の主張がこれらの原則に抵触しないかを最初にチェックせよ」
- **確信度スケールの運用ルールを具体化する**: 「50%（方向性OK、裏付け不十分）が最頻値であり、90% を付与するのは年に1-2回のみ。70% 以上を付与する際は、必ず定量的裏付けを引用せよ」
- **却下基準の即座適用を指示する**: 「rule01（能力 ≠ 結果）違反が検出された場合、それ以上の分析を進めず、即座に確信度を30%以下にセットせよ」

#### KB1（8ルール）→ Evaluation Checklist Skill

KB1 の8ルールは演繹的チェックリストであり、Numerai SKILLS.md の「特徴量テスト手法」に相当する。全ての主張に対して機械的に確認すべき項目のリストであり、SKILLS.md 化は比較的直接的に行える。

変換のポイントは、各ルールに**具体的なトリガー条件と判定手順**を付加することである。現在の KB1 は「能力と結果を区別せよ」という抽象的な表現だが、SKILLS.md 化では「主張のテキストに『市場シェア』『売上高』『利益率の水準』等の結果指標が含まれる場合、それが原因（能力）ではなく結果であることを検証せよ。原因に遡れない場合は rule01 違反としてフラグを立てよ」のように、AI が機械的に適用可能な形式にする。

#### KB2（12パターン）→ Failure/Success Diagnosis Skill

KB2 は帰納的パターンカタログであり、Numerai SKILLS.md の「失敗の診断パターン」に直接対応する。却下パターン A-G は「この種の主張は典型的に過大評価される」という失敗パターンであり、高評価パターン I-V は「この種の主張は構造的に頑健である」という成功パターンである。

SKILLS.md 化のポイントは、**パターンマッチングの優先順位と組み合わせルール**を明示することである。現在の KB2 は個々のパターンを独立に記述しているが、実際の判断では複数のパターンが同時に該当する場合がある。「パターン A（業界共通の成長要因）とパターン I（定量的差別化）が同時に該当する場合、I の方が優先される」のような優先順位ルールを追加する。

#### KB3（5社34件）→ Few-Shot Calibration Skill

KB3 は5社34件の具体的な判断例であり、Numerai SKILLS.md の「結果の評価方法」における較正基準に相当する。AI エージェントが自分のスコアリングを KB3 の事例と比較し、系統的な偏り（過信・過小評価）がないかを自己較正するために使用する。

SKILLS.md 化のポイントは、**較正プロセスを明示的なステップとして定義する**ことである。「スコアリング完了後、KB3 の同業種・類似主張の事例と比較し、自分のスコアが Y の判断から ±15% 以上乖離している場合は、乖離の理由を記述せよ。理由が説明できない場合は、Y の判断に合わせて調整せよ」。

### 5.3 MCP によるエージェント自律化の設計

Numerai が MCP で実現した「エージェントの自律リサーチループ」は、当プロジェクトの MAS アーキテクチャに直接適用できる。現在の MAS 設計は、Python バックテストエンジンが各フェーズを順次呼び出す「オーケストレーション型」だが、Numerai パターンに倣えば、エージェント自身が MCP を通じてデータを取得し、分析を実行し、結果を評価する「自律型」に進化させることが可能である。

具体的な MCP 統合設計を以下に示す。

**SEC EDGAR MCP の自律的利用。** 現在は Python の `PointInTimeDataManager` が SEC Filings を取得し、エージェントに渡している。Numerai パターンでは、Fundamental Analyst エージェント自身が SEC EDGAR MCP を直接呼び出し、必要な Filings を自律的に取得する。エージェントは SKILLS.md（dogma + KB1）に基づいて「何を見るべきか」を知っており、MCP を通じて「それを取得する」能力を持つ。

```
[Fundamental Analyst エージェント]
    |
    |-- Skills: dogma要約 + KB1 チェックリスト
    |
    |-- MCP: sec-edgar-mcp
    |   |-- get_recent_filings(ticker, form="10-K")
    |   |-- get_filing_sections(filing, sections=["Item1", "Item7"])
    |   |-- get_financials(ticker, metrics=["roe", "operating_margin"])
    |   |-- get_key_metrics(ticker)
    |
    |-- 出力: stock_analysis.json（thesis_points + risk_factors + evidence）
```

**市場データ MCP の自律的利用。** yfinance データ取得を MCP 化することで、Valuation Analyst や Macro Analyst が自律的に価格データ、時価総額、リターンデータを取得できるようになる。

**判断結果の自律的記録。** エージェントが自律的に `decision_log.json` を生成し、ファイルシステムに書き込む。Python バックテストエンジンはこのファイルを読み取ってパフォーマンス計算を行う。

### 5.4 「研究者の暗黙知を構造化する」アプローチの深化

Numerai の SKILLS.md アプローチから得られる最も深い洞察は、**暗黙知の構造化は一度きりのプロジェクトではなく、継続的なプロセスである**ということだ。

当プロジェクトの KB は、アナリスト Y の判断パターンを構造化したものであり、現在は5社34件の事例に基づいている。しかし、MAS がバックテストを実行し、その結果を Y がレビューするたびに、新たな暗黙知が発見される。「この銘柄で AI は70%の確信度を付けたが、Y は30%だった。その理由は、AI が rule07（純粋競合差別化）を正しく適用できていなかったからだ」といった発見は、KB の改善に直結する。

Numerai はこの継続的改善ループを、プラットフォームの設計に組み込んでいる。コントリビューターの予測が Meta Model に統合され、そのパフォーマンスが測定され、成功/失敗のパターンが SKILLS.md にフィードバックされる。当プロジェクトでも同様のループを設計する必要がある。

```
[暗黙知拡充ループ]

1. MAS がバックテストを実行
    |
    v
2. 各リバランス日の decision_log.json を生成
    |
    v
3. アナリスト Y がレビュー（特に大きな乖離がある銘柄を重点的に）
    |
    v
4. 乖離の原因を分析
    |-- KB1 ルールの適用ミス → KB1 の記述を改善
    |-- 新しいパターンの発見 → KB2 に追加
    |-- 判断の較正ずれ → KB3 に新しい事例を追加
    |-- dogma の原則に反する判断 → dogma の記述を明確化
    |
    v
5. 更新された KB で MAS を再実行
    |
    v
6. 1. に戻る
```

このループの1サイクルあたりのコストは、Numerai の場合はプラットフォームが自動的に処理するが、当プロジェクトの場合は Y の人的レビューが必要である。この人的レビューのコストを削減するため、「AI 批判エージェント」（CA-Eval で実績のある ca-claim-extractor / ca-pattern-verifier パターン）を活用し、Y が確認すべき箇所を自動的に絞り込む仕組みが有効である。

### 5.5 自律的アルファ探索ループの設計

Numerai の「The Era of Agents」ビジョンを当プロジェクトに翻訳すると、「MAS が自律的にアルファ（超過リターン）の源泉を探索するループ」の設計になる。

現在の MAS 設計（`2026-02-16_mas-investment-team-poc-design.md`）は、以下の線形パイプラインを前提としている。

```
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5
（破綻分析→スクリーニング→分析→ディベート→判断→ポートフォリオ構築）
```

Numerai パターンに倣った自律的アルファ探索ループは、これに反復的な仮説検証サイクルを追加する。

```
[自律的アルファ探索ループ]

Outer Loop（四半期ごと: バックテストエンジンが制御）
    |
    Inner Loop（エージェントが自律実行）
    |
    |-- 1. 仮説生成
    |   - Skills（dogma + KB）に基づく投資仮説の構造化
    |   - 前回リバランスの結果からのフィードバック反映
    |
    |-- 2. データ収集（MCP 経由）
    |   - SEC EDGAR MCP: Filings、財務データ
    |   - 市場データ: 価格、リターン、時価総額
    |
    |-- 3. 分析（4並列エージェント）
    |   - Fundamental: KB1 チェックリストベースの評価
    |   - Valuation: DCF + 相対バリュエーション
    |   - Sentiment: トランスクリプト + KB1-T/KB3-T
    |   - Macro: レジーム判定
    |
    |-- 4. CA 評価（CA Evaluator）
    |   - dogma + KB1 + KB2 + KB3 全量参照
    |   - 各主張に対するルール適用結果を構造化出力
    |
    |-- 5. ディベート（Bull vs Bear）
    |   - KB2 パターンに基づく論拠構築
    |   - Round 1 → Round 2 の反復
    |
    |-- 6. 判断（Fund Manager）
    |   - dogma 確信度スケール準拠
    |   - 2軸スコアリング（CA確信度 + 投資確信度）
    |
    |-- 7. 自己診断
    |   - KB3 事例との比較による較正チェック
    |   - KB2 パターンとのマッチング確認
    |   - 大きな乖離がある場合は 3. に戻って再分析
    |
    |-- 8. ポートフォリオ構築
    |   - 確信度加重 + リスク制約
    |   - decision_log.json + portfolio.json 出力
```

ステップ7の「自己診断」が、Numerai の SKILLS.md における「反復プロセス」に相当する。エージェントが自分の判断を KB3 の事例と比較し、系統的な偏りを検出した場合に自律的に再分析を実行する。これにより、単一パスの線形パイプラインでは見逃される判断の歪みを補正できる。

### 5.6 Numerai の「40時間で Millennium のコードを再現」への示唆

Craib の「Millennium の年間1億ドルのポッドのコードが、Claude で40時間で再現できる」という主張は、当プロジェクトにとって2つの示唆を持つ。

**第一の示唆: Skills/KB が差別化の核心である。** コードそのものは AI が短時間で生成できる。しかし、「何を」「どのように」分析すべきかという投資哲学と判断基準（当プロジェクトでは dogma + KB）は、人間の経験と判断に基づいて構築される必要があり、これが真の差別化要因である。Numerai の SKILLS.md が、コード以上に価値があるのと同じ構造である。

当プロジェクトのナレッジベースは、アナリスト Y の12社での評価実績に基づく8ルール + 12パターン + 34件の具体例で構成されている。これは「アナリスト Y のクローン」を作るための暗黙知であり、AI が自力で生成することはできない。Millennium のポッドのコードは再現可能かもしれないが、ポッドマネージャーの20年の経験から来る市場直感や、特定のセクターにおける暗黙の判断基準は、SKILLS.md/KB として構造化しなければ AI には伝達できない。

**第二の示唆: スケーラビリティの革命。** 1人のアナリスト Y の暗黙知を KB として構造化すれば、その KB を持つ AI エージェントを無限にスケールできる。Millennium が1つのポッドに年間1億ドルを支払っているのは、そのポッドマネージャーの知識と判断力がスケールしないからである。KB として構造化された暗黙知は、AI エージェントの数だけスケールする。

具体的には、当プロジェクトの KB を持つ CA Evaluator エージェントは、1銘柄の評価に数分を要するが、これを50銘柄に並列展開しても、所要時間はほぼ変わらない（Agent Teams の並列実行機能）。人間のアナリスト Y が50銘柄を評価するには数週間を要するが、AI エージェントは同じ品質の評価を数時間で完了できる。

## 6. 結論と戦略的示唆

### 6.1 Numerai から学ぶべき3つの原則

**原則1: Skills First, Code Second.** Numerai が示した最大の教訓は、AI エージェントの性能を決定するのはコードやモデルアーキテクチャではなく、Skills（暗黙知の構造化ドキュメント）であるということだ。当プロジェクトの dogma + KB は既にこの原則を体現しているが、「AI エージェントが読み込んだ瞬間に行動を開始できる形式」への変換（SKILLS.md 化）が次のステップである。

**原則2: MCP で自律性を解放する。** Skills が「何をすべきか」を定義し、MCP が「それをどう実行するか」のインターフェースを提供する。この2つの組み合わせにより、完全な自律リサーチループが実現する。当プロジェクトは既に SEC EDGAR MCP を使用しているが、エージェント自身が MCP を直接呼び出す自律型アーキテクチャへの移行が必要である。

**原則3: 継続的な暗黙知拡充ループを設計する。** Skills/KB は静的なドキュメントではなく、バックテスト結果と人間のフィードバックに基づいて継続的に改善されるべき「生きたナレッジベース」である。Numerai はこのループをプラットフォームの設計に組み込んでいる。当プロジェクトも、Y のレビューを効率化する仕組み（AI 批判エージェントによる絞り込み）と、KB バージョニングシステムを整備する必要がある。

### 6.2 当プロジェクトの独自性

Numerai と当プロジェクトは、アプローチの共通点が多いが、以下の点で本質的に異なる。

| 項目 | Numerai | 当プロジェクト（MAS4InvestmentTeam） |
|------|---------|-------------------------------------|
| **目的** | 統計的予測の精度最大化 | 説明可能な投資判断の構築 |
| **出力** | 数値予測（株価の相対順位） | 構造化された投資根拠（decision_log.json + FM レポート） |
| **暗黙知の性質** | データサイエンスの方法論 | アナリストの投資判断基準 |
| **透明性** | Meta Model のブラックボックス性 | 全判断の完全な監査トレース |
| **スケール** | 数千人のコントリビューター | 1人のアナリスト Y の知識のスケール |

当プロジェクトの独自性は、「説明可能性」と「アナリスト固有の判断基準の再現」にある。Numerai が「最も予測精度が高いモデル」を追求するのに対し、当プロジェクトは「なぜその投資判断をしたのか」を人間が理解・検証可能な形で記録することを重視する。この違いは、規制環境（金融庁、SEC）や投資家への説明責任という実務的要請から来ている。

### 6.3 今後の展望

Numerai の「The Era of Agents」宣言は、当プロジェクトの方向性を強力に裏付けるものである。世界最大級のクオンツプラットフォームが「AI エージェントの時代」を宣言し、Skills + MCP + 自律リサーチループのフレームワークを構築していることは、当プロジェクトが追求する「AI マルチエージェントによる投資チーム」というコンセプトの実現可能性と市場適合性を示している。

当面の最優先事項は以下の3つである。

1. **KB の SKILLS.md 化**（Stage 3 の追加タスク 3.8-3.11）: アナリスト Y の暗黙知を、AI エージェントが即座に行動に移せる形式に変換する
2. **CA Strategy Phase 1 のブロッカー解消**: 自律リサーチループの前提として、LLM による主張抽出（当プロジェクト版の Faith データセット特徴量生成）が正常に動作する必要がある
3. **暗黙知拡充ループの設計**: バックテスト結果 → Y レビュー → KB 更新 → 再実行のサイクルを、AI 批判エージェントで半自動化する

Numerai が示した道筋は明確である。AI エージェントの時代において、競争優位性の源泉はコードやモデルではなく、「暗黙知の構造化」と「自律的な実行フレームワーク」にある。当プロジェクトは、この原則を投資判断の説明可能性という独自の軸で実現することを目指す。
